# Transformer Hawkes Process: A From-Scratch Implementation

A from-scratch PyTorch implementation of the Transformer Hawkes Process (Zuo et al., 2020), built to understand temporal point process modelling at the implementation level rather than through a pre-built library. The repository includes multi-head self-attention written without `nn.MultiheadAttention`, three temporal encoding strategies for irregularly-timed event sequences, a multivariate Hawkes process simulator used to generate data with a known ground-truth intensity function, and the full THP training pipeline with a maximum-likelihood loss.

It also includes a data leakage bug that was found, diagnosed, and fixed during development, documented in full in the notebook and in the test suite rather than quietly removed from the history. More on that below, because it ended up being the most useful part of the exercise.

## Why build this from scratch

Point process models combine a few things that are each individually easy to get subtly wrong: attention over irregularly-spaced sequences, continuous-time embeddings, and a likelihood function whose normalising term has no closed form and has to be estimated. Calling a library function hides exactly the details that are worth understanding. Writing each component by hand, and checking it against either a known mathematical reference (PyTorch's own attention kernel, a closed-form stationary rate) or a synthetic dataset with a known answer, was the point of the exercise.

## Running it

```bash
pip install -r requirements.txt
./run_tests.sh                     # runs all 23 tests across 4 modules
jupyter notebook notebooks/thp_implementation.ipynb
```

The notebook runs end-to-end on CPU in a few minutes. No GPU is required at this scale.

## The bug, and why it's left in

The first working version of the training loop used the Transformer's hidden state at position *i* to predict event *i*'s own type and intensity. This looks reasonable at first glance, because the self-attention is correctly causal: position *i* can only attend to positions 1 through *i*, never anything later. But causal attention only restricts what a position can *look at*; it doesn't strip out the information already baked into that position's own input embedding before attention even runs. Position *i*'s embedding starts from event *i*'s own type and timestamp, and that information survives through the residual connections regardless of what attention does on top. So using hidden state *i* to predict event *i* isn't really prediction — it's closer to reading the answer off the input.

This stayed invisible until I checked the trained model's validation log-likelihood against the log-likelihood of the *true* Hawkes process that generated the validation data — something only possible because the data was synthetic and the ground truth was known. The model scored *better* than the process that actually generated the data, which isn't something a correctly specified model can do in expectation. That mismatch was the signal that something was leaking, and tracing it back led to the fix: shift everything by one position. Event *i* is predicted from hidden state *i − 1*, never hidden state *i*. After the fix, the model's validation NLL sat just above the true process's NLL, which is the relationship you'd actually expect.

Two things about this felt worth keeping visible rather than tidying away. First, the fix is genuinely simple, which makes it a particularly good example of how an architecturally "correct" causal mask can still coexist with leakage somewhere else in the pipeline — the bug wasn't in the attention mechanism at all, it was in how the intensity head was wired up afterwards. Second, `tests/test_thp_model.py` now includes a dedicated test, `test_event_intensity_does_not_leak_own_type_or_time`, that perturbs a single event's own type and timestamp and checks that the hidden state used to predict it is provably unaffected. That test exists specifically so this exact bug can't quietly reappear.

## Validating against ground truth

Real event data never comes with a known intensity function, which makes it hard to tell whether a point process model has actually learned the right thing or just produces plausible-looking numbers. The notebook works around this by simulating data from a multivariate Hawkes process with known parameters, training the model on it, and then comparing the model's predicted intensity against the *exact* mathematical intensity on held-out sequences the model never saw during training.

The result is moderate rather than impressive: a Pearson correlation around 0.55–0.60 between true and predicted intensity, with the model correctly identifying *when* intensity spikes after an excitation event but generally underestimating the height of the spike. That's a realistic outcome for a model trained on a few thousand events with no hyperparameter search, and it's reported as-is rather than tuned until the number looks better — a moderate, honestly-measured result is worth more than a suspiciously clean one.

## Limitations

This is a learning and validation exercise, not a production system or a research contribution. In particular: the training data is synthetic, not real-world; the dataset is small (a few thousand events) compared to the tens of thousands of sequences used in the original THP paper; the integral term in the likelihood is estimated by Monte Carlo and carries some irreducible variance at the sample size used here; there is no hyperparameter search or comparison against baseline models; and the leakage bug described above was caught for this specific architecture's wiring, not via a general-purpose leakage detector that would automatically catch the equivalent mistake in a different model. The notebook's final section goes through these in more detail.

## A note on the data

Although nothing here touches real personal data, the broader class of model this repository explores — sequence models over individual behavioural histories — is increasingly applied to genuinely sensitive data: insurance claims, financial transactions, healthcare visits. Worth naming plainly: this kind of data normally requires a clear legal basis and informed consent to process; models trained on historical interaction data can encode and perpetuate existing biases in who was offered what, which matters a great deal if the output ever touches pricing; and a long enough individual event history starts to function less like a feature vector and more like a fingerprint, which raises re-identification risks that don't apply to simpler cross-sectional models.
